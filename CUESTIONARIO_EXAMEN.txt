================================================================================
bloque 1 - hmac & jwt
================================================================================

pregunta 1
-----------
que hace la anotacion @PostConstruct en JwtTokenProvider y por que se usa ahi?

respuesta:
se ejecuta una sola vez despues de que spring crea el bean. en este proyecto
decodifica la secretKey de base64 a bytes:
    secretKeyByte = Base64.getDecoder().decode(secretKey);
se hace ahi para no repetir la decodificacion en cada llamada a createToken o validateToken.

---

pregunta 2
-----------
que es un claim en un jwt y cuales se usan en este proyecto?

respuesta:
un claim es un dato que va dentro del payload del jwt (la parte del medio).
en este proyecto se usan:
    .subject(user.getUsername())   -> el username del usuario
    .id(user.getId().toString())   -> el id unico del usuario
    .issuedAt(new Date())          -> cuando se creo el token
    .expiration(validity)          -> cuando expira

---

pregunta 3
-----------
que hace Keys.hmacShaKeyFor(secretKeyByte) y para que se usa?

respuesta:
convierte un arreglo de bytes en un objeto SecretKey que entiende la libreria jjwt.
se usa dos veces:
    - en createToken() para firmar el jwt con hmac-sha256
    - en validateToken() para verificar la firma del token recibido

---

pregunta 4
-----------
cual es la diferencia entre signWith() en createToken y verifyWith() en validateToken?

respuesta:
    signWith(secretKey)  -> al crear el token, jjwt calcula el hmac del header+payload
                           y lo agrega como la tercera parte del token (la firma)
    verifyWith(secretKey) -> al validar, jjwt recalcula el hmac y lo compara con la firma
                            que trae el token. si no coinciden lanza JwtException

---

pregunta 5
-----------
que hace resolveToken y como sabe si el header es valido?

respuesta:
extrae el token del header Authorization. verifica que empiece con "Bearer ":
    if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
        return bearerToken.substring(7); // corta "Bearer " (7 caracteres) y retorna solo el token
    }
si no empieza con eso retorna null.

---

pregunta 6
-----------
que excepciones puede lanzar validateToken y que significa cada una?

respuesta:
    ExpiredJwtException         -> el token es valido pero ya expiro (vencio la fecha)
    JwtException                -> la firma no coincide o el token esta malformado
    IllegalArgumentException    -> el token es null o vacio
    UsernameNotFoundException   -> el token es valido pero el usuario ya no existe en db

---

pregunta 7
-----------
que hace JwtTokenFilter y cuando se ejecuta?

respuesta:
se ejecuta en CADA request http. extiende OncePerRequestFilter para garantizar
que se ejecuta una sola vez por request. su flujo es:
    1. extrae el token del header Authorization via resolveToken()
    2. si token es null, deja pasar el request (puede ser endpoint publico)
    3. si hay token, llama validateToken()
    4. si es valido, setea SecurityContextHolder con el usuario autenticado
    5. si falla (expirado, invalido) retorna 401 UNAUTHORIZED

---

pregunta 8
-----------
que es WebSocketAuthChannelInterceptor y cuando actua?

respuesta:
implementa ChannelInterceptor y intercepta mensajes STOMP antes de que lleguen
al broker. solo actua cuando el comando es CONNECT (la primera conexion websocket).
valida el jwt del header Authorization del frame STOMP. si el token es invalido
lanza IllegalArgumentException y rechaza la conexion.

---

pregunta 9
-----------
por que el endpoint /api/v1/stereum/outbound esta como permitAll() en security
si recibe datos importantes?

respuesta:
porque stereum es un servicio externo que no tiene un jwt de tu sistema.
no puede autenticarse como usuario. la seguridad se hace via HMAC: se verifica
que el request viene de stereum comparando la firma del body con X-Signature.
si la firma no coincide se rechaza el request manualmente.

---

pregunta 10
-----------
que valor tiene validityInMinutes y en que unidad esta realmente?

respuesta:
el valor por defecto es 28800000. a pesar de que el campo se llama validityInMinutes,
el valor es en milisegundos. 28800000 ms = 8 horas.
se configura en application.properties con:
    security.jwt.token.expire-length=28800000


================================================================================
bloque 2 - jsonobject
================================================================================

pregunta 11
-----------
de que libreria viene JSONObject y como se agrega al proyecto?

respuesta:
viene de la libreria org.json. se agrega en pom.xml:
    <dependency>
        <groupId>org.json</groupId>
        <artifactId>json</artifactId>
        <version>20240303</version>
    </dependency>

---

pregunta 12
-----------
cual es la diferencia entre JSONObject y JSONArray?

respuesta:
    JSONObject  -> representa un objeto json: { "clave": "valor" }
    JSONArray   -> representa un arreglo json: [ "item1", "item2" ]
en el proyecto se usa JSONObject para construir los bodies de los requests a stereum.

---

pregunta 13
-----------
en PagoController, como se convierte un Map a JSONObject y por que se hace asi?

respuesta:
    new JSONObject(body)   // body es Map<String, Object>
el constructor de JSONObject acepta un Map y lo convierte automaticamente.
se usa porque el endpoint recibe @RequestBody Map<String, Object>, lo que permite
recibir cualquier estructura json sin definir una clase especifica.

---

pregunta 14
-----------
como se anida un JSONObject dentro de otro? mostrar con el ejemplo del proyecto.

respuesta:
se crea el objeto hijo primero y luego se pone como valor del padre:
    JSONObject customer = new JSONObject();
    customer.put("name", "Juan");
    customer.put("lastname", "Laredo");

    JSONObject req = new JSONObject();
    req.put("country", "BO");
    req.put("customer", customer);   // JSONObject dentro de JSONObject

---

pregunta 15
-----------
como se convierte un JSONObject a String para mandarlo como body de un request?

respuesta:
con el metodo .toString():
    restClient.post()
        .body(request.toString())   // JSONObject -> String json
jjwt necesita el body como String para enviarlo por http.


================================================================================
bloque 3 - stereum
================================================================================

pregunta 16
-----------
cuales son los dos endpoints de StereumController y que hace cada uno?

respuesta:
    POST /api/v1/stereum/charge    -> inicia el cobro. recibe el orderId, construye
                                      el JSONObject, llama a SistemaB2B.callStereum()
                                      y retorna el QR y el link de pago.

    POST /api/v1/stereum/outbound  -> webhook. stereum llama a este endpoint cuando
                                      el pago cambia de estado. verifica la firma hmac
                                      y notifica al frontend por websocket.

---

pregunta 17
-----------
que campos lleva el JSONObject que se manda a stereum en el endpoint /charge?

respuesta:
    country                  -> "BO" (Bolivia)
    amount                   -> "100"
    currency                 -> "USDT"
    network                  -> "POLYGON"
    charge_reason            -> "COMPRA A: {nombre del proveedor}"
    idempotency_key          -> ordenCompra.getId().toString() (UUID de la orden)
    reservation_validity_time-> "10"
    customer                 -> JSONObject con name, lastname, document_number

---

pregunta 18
-----------
que es el idempotency_key y para que sirve en el flujo de pago?

respuesta:
es el UUID de la OrdenCompra. sirve para que cuando stereum manda el webhook,
puedas identificar a que orden corresponde ese pago. tambien evita procesar
el mismo pago dos veces (idempotencia = misma operacion con mismo key = mismo resultado).

---

pregunta 19
-----------
que header usa SistemaB2B para autenticarse con la API de stereum?

respuesta:
    .header("x-api-key", stereumToken)
el valor viene de application.properties: stereum.api.key

---

pregunta 20
-----------
que campos trae StereuemApiResponse cuando stereum responde al /charge?

respuesta:
    amount, currency, network, id
    qrBase64       -> imagen qr en base64 para mostrar al usuario
    paymentLink    -> link alternativo de pago
    transactionStatus
    collectingAccount
    expirationTime

---

pregunta 21
-----------
que campos trae StereumPayResponse cuando stereum llama al webhook?

respuesta:
    notificationType  -> tipo de notificacion
    id                -> id de la notificacion
    timestamp         -> cuando ocurrio
    transaction       -> objeto Transaction con el detalle del pago

y dentro de Transaction:
    status, statusDescription, amount, amountReceived
    currency, network, country
    idempotencyKey (el UUID de la orden), id, createdDate, paymentDate


================================================================================
bloque 4 - webhook
================================================================================

pregunta 22
-----------
que es un webhook y como se diferencia de una llamada normal a una api?

respuesta:
un webhook es un endpoint tuyo que un servicio externo llama cuando ocurre un evento.
en una api normal vos llamas al servicio para pedir datos (pull).
con un webhook el servicio te llama a vos cuando tiene algo nuevo (push).
en el proyecto: vos llamas a stereum para crear el cobro, pero stereum te llama
a vos cuando el pago se confirma.

---

pregunta 23
-----------
por que el body del webhook se recibe como String y no directamente como StereumPayResponse?

respuesta:
porque para calcular el hmac necesitas el body exactamente como llego, sin
ningun cambio de formato. si jackson lo deserializara primero podria reordenar
campos o cambiar espacios, haciendo que el hmac calculado no coincida con el
que mando stereum.

---

pregunta 24
-----------
que pasaria si no se verificara el hmac en el webhook?

respuesta:
cualquiera podria mandar un request falso a /api/v1/stereum/outbound diciendo
que una orden fue pagada cuando en realidad no lo fue. el sistema marcaria
ordenes como pagadas sin que realmente hayan sido procesadas.

---

pregunta 25
-----------
como llega la notificacion del webhook al frontend finalmente?

respuesta:
via websocket. despues de verificar el hmac:
    template.convertAndSend(
        "/paymenting/" + response.getTransaction().getIdempotencyKey(),
        response.getTransaction().getStatus()
    );
el frontend tiene que estar suscrito al topic "/paymenting/{idempotencyKey}"
para recibir el status en tiempo real sin hacer polling.


================================================================================
bloque 5 - transactions & metodos asincronos
================================================================================

pregunta 26
-----------
que garantiza @Transactional en un metodo de servicio?

respuesta:
que todas las operaciones de base de datos dentro del metodo se ejecuten en
una sola transaccion. si algo falla, se hace rollback de todo lo que se hizo
en esa transaccion. si todo va bien, se hace commit al final.

---

pregunta 27
-----------
cual es la diferencia entre @Transactional y @Transactional(readOnly = true)?

respuesta:
    @Transactional              -> permite leer y escribir en db. puede bloquear registros.
    @Transactional(readOnly=true) -> solo lectura. hibernate puede optimizar las consultas,
                                    no hace flush, no trackea cambios. es mas eficiente
                                    para metodos que solo consultan datos.

---

pregunta 28
-----------
que hace Propagation.REQUIRES_NEW y por que se usa en LogService?

respuesta:
suspende la transaccion actual (si la hay) y crea una nueva transaccion independiente.
se usa en LogService porque aunque el metodo que llama a logService.info() falle
y haga rollback, el log YA fue guardado en su propia transaccion y no se revierte.
asi nunca se pierde el registro del error.

---

pregunta 29
-----------
que hace @EnableAsync y donde esta en el proyecto?

respuesta:
activa el soporte para metodos asincronos en spring.
esta en la clase principal B2BProyectApplication (o en la configuracion principal).
sin esta anotacion, @Async no tiene efecto y los metodos se ejecutan de forma sincrona.

---

pregunta 30
-----------
que es el ThreadPoolTaskExecutor llamado "taskLog" y como esta configurado?

respuesta:
es un pool de hilos dedicado a ejecutar los metodos asincronos de LogService.
configurado en InjectConfiguration.java:
    corePoolSize   -> 5  (hilos siempre activos, configurable con async.core-pool-size)
    maxPoolSize    -> 10 (maximo de hilos si la cola se llena, async.max-pool-size)
    queueCapacity  -> 20 (cuantas tareas pueden esperar en cola, async.queue-capacity)
    threadNamePrefix -> "Miguel-"  (los hilos se llaman Miguel-1, Miguel-2, etc.)

---

pregunta 31
-----------
cuando se usa @Async("taskLog"), quien ejecuta el metodo y que pasa con el hilo original?

respuesta:
spring toma el metodo y lo pone en la cola del ThreadPoolTaskExecutor "taskLog".
uno de los hilos del pool (Miguel-1, Miguel-2, etc.) lo ejecuta en segundo plano.
el hilo original que hizo la llamada retorna inmediatamente sin esperar el resultado.

---

pregunta 32
-----------
que pasa si el pool esta lleno y la cola tambien cuando se llama un metodo @Async?

respuesta:
el ThreadPoolTaskExecutor rechaza la tarea. por defecto lanza una excepcion
(TaskRejectedException). en el proyecto esto ocurriria si mas de 10 hilos
estan ocupados y hay mas de 20 tareas en cola esperando.

---

pregunta 33
-----------
que tiene de especial el caso noRollbackFor = NotDataFoundException.class en EmpresaService?

respuesta:
por defecto @Transactional hace rollback ante cualquier RuntimeException.
con noRollbackFor le dices a spring que si ocurre NotDataFoundException
NO haga rollback de la transaccion. util cuando queres guardar algo aunque
no se haya encontrado un dato (por ejemplo, registrar el intento igualmente).


================================================================================
bloque 6 - logs
================================================================================

pregunta 34
-----------
cuales son los dos tipos de logging en el proyecto y en que se diferencian?

respuesta:
    tipo 1 - @Slf4j (logging a consola/archivo):
        usa lombok para generar el logger automaticamente.
        los mensajes van a la consola y/o archivo de log de spring.
        no se guardan en base de datos.

    tipo 2 - LogService (logging persistente a db):
        guarda cada log en la tabla "log" de la base de datos.
        es asincrono: no bloquea al que lo llama.
        tiene niveles: INFO, WARN, ERROR, DEBUG, TRACE.

---

pregunta 35
-----------
que genera la anotacion @Slf4j de lombok?

respuesta:
genera automaticamente en la clase la siguiente linea:
    private static final Logger log = LoggerFactory.getLogger(MiClase.class);
asi podemos usar log.info(), log.error(), etc. sin declarar el logger manualmente.

---

pregunta 36
-----------
como se usa el placeholder {} en los mensajes de log?

respuesta:
    log.info("usuario autenticado: {}", user.getEmail());
el {} se reemplaza con el valor del argumento siguiente. es mas eficiente que
concatenar strings porque si el nivel de log esta desactivado, ni siquiera
evalua el toString() del objeto.

---

pregunta 37
-----------
que campos tiene la entidad Log en base de datos?

respuesta:
    id          -> String, generado por UUID (@UuidGenerator)
    level       -> enum LogLevel (INFO, WARN, ERROR, DEBUG, TRACE), columna "_level"
    descripcion -> String de hasta 4000 caracteres
    + los campos de AuditableEntity (createdAt, updatedAt, createdBy, etc.)

---

pregunta 38
-----------
como se llama a LogService desde EmpresaService? que ocurre en ese momento?

respuesta:
    logService.info("Creando empresa nueva. Vamos!");
en ese momento spring intercepta la llamada, la pone en la cola del pool "taskLog"
y retorna inmediatamente. el hilo de EmpresaService continua con el save() de la empresa.
en paralelo, un hilo "Miguel-X" guarda el log en db en su propia transaccion.


================================================================================
bloque 7 - jobs
================================================================================

pregunta 39
-----------
que anotacion habilita los jobs en spring y donde esta en el proyecto?

respuesta:
    @EnableScheduling
esta en la clase principal B2BProyectApplication. sin ella, @Scheduled no tiene efecto.

---

pregunta 40
-----------
cual es el unico job que existe en el proyecto y que hace?

respuesta:
esta en EmpresaService:
    @Scheduled(cron = "0 */1 * * * *")
    public void listarEmpresas() {
        List<EmpresaDTO> empresas = empresaRepository.findAllDTO();
        log.info("Listando empresas: {}", empresas);
    }
lista todas las empresas de la db y las imprime en el log. se ejecuta cada minuto.

---

pregunta 41
-----------
explica la expresion cron "0 */1 * * * *" campo por campo.

respuesta:
formato de spring: segundos minutos horas dia-mes mes dia-semana

    0      -> en el segundo 0 de cada minuto
    */1    -> cada 1 minuto (el */ significa "cada")
    *      -> cualquier hora
    *      -> cualquier dia del mes
    *      -> cualquier mes
    *      -> cualquier dia de la semana

resultado: se ejecuta al segundo 0 de cada minuto, todos los dias.

---

pregunta 42
-----------
como se escribiria un cron para ejecutar algo todos los dias a las 8am?

respuesta:
    @Scheduled(cron = "0 0 8 * * *")
    -> segundo 0, minuto 0, hora 8, cualquier dia, cualquier mes, cualquier dia semana

---

pregunta 43
-----------
el job listarEmpresas usa @Transactional? por que si o por que no?

respuesta:
no tiene @Transactional. llama a empresaRepository.findAllDTO() directamente.
al no tener @Transactional, spring puede abrir una transaccion de solo lectura
por defecto dependiendo de la configuracion del repositorio.
si necesitara garantias transaccionales se le agregaria @Transactional(readOnly = true).

---

pregunta 44
-----------
que diferencia hay entre @Scheduled y @Async?

respuesta:
    @Scheduled -> spring llama al metodo automaticamente segun el cron o el intervalo.
                  nadie lo invoca manualmente. se ejecuta en un hilo de scheduler.

    @Async     -> vos llamas al metodo manualmente desde el codigo, pero spring
                  lo ejecuta en un hilo del ThreadPool en lugar del hilo actual.
                  el llamador no espera el resultado.

preguntas bonus - integracion de temas
================================================================================

pregunta 45
-----------
describe el flujo completo de un pago con stereum desde que el usuario hace click
hasta que ve la confirmacion en pantalla.

respuesta:
    1. frontend llama POST /api/v1/stereum/charge con { orderId }
    2. el request pasa por JwtTokenFilter que valida el JWT del usuario
    3. StereumController busca la OrdenCompra en db
    4. construye un JSONObject con los datos del pago
    5. SistemaB2B.callStereum() hace POST a stereum con x-api-key
    6. stereum responde con QR (base64) y payment link
    7. el usuario escanea el QR y paga en su billetera cripto
    8. stereum llama POST /api/v1/stereum/outbound (el webhook)
    9. StereumController calcula HMAC del body y compara con X-Signature
    10. si firma valida, deserializa el body a StereumPayResponse
    11. template.convertAndSend() manda el status por WebSocket
    12. el frontend que estaba suscrito a /paymenting/{idempotencyKey} recibe el status
    13. la UI se actualiza en tiempo real sin que el usuario recargue la pagina

---

pregunta 46
-----------
por que LogService usa REQUIRES_NEW junto con @Async? no seria suficiente con solo @Async?

respuesta:
con solo @Async el metodo se ejecuta en otro hilo, pero puede participar en
la transaccion del hilo que lo llamo (si la hay). con REQUIRES_NEW se garantiza
que el log se guarda en su PROPIA transaccion independiente.
caso critico: si el metodo que llamo a logService.error() falla y hace rollback,
con solo @Async el log tambien podria revertirse. con REQUIRES_NEW el log
se guardo en una transaccion ya confirmada y no se pierde.

---

pregunta 47
-----------
que pasaria si se quitara @EnableWebSecurity de WebSecurityConfiguration?

respuesta:
spring no aplicaria ninguna regla de seguridad. todos los endpoints quedarian
publicos sin necesidad de jwt. el JwtTokenFilter seguiria existiendo como bean
pero no se agregaria a la cadena de filtros de seguridad, por lo que no se
ejecutaria en ninguna request.
