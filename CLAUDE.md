## Build & Run Commands

```bash
# Build all modules
mvn clean install

# Run the application (default profile)
mvn spring-boot:run -pl B2B-api

# Run with prod profile
mvn spring-boot:run -pl B2B-api -Dspring-boot.run.arguments="--spring.profiles.active=prod"

# Package JAR
mvn clean package

# Package JAR for prod (excludes logback.xml and application*.properties)
mvn clean package -Denv=prod

# Run all tests
mvn test

# Run a single test class
mvn test -pl B2B-api -Dtest=MyTestClass

# Start the database (Docker)
docker compose -f extra/docker-compose.yaml up -d
```

## Project Structure

Three-module Maven project (Java 21, Spring Boot 4.0.6):

- **`data/`** — JPA entities, repositories, DTOs, projections, enums. No business logic here.
- **`core/`** — Services, external API integrations (`integracion/`), JWT provider, async logging.
- **`B2B-api/`** — Controllers, Spring Security/WebSocket/DI config, application entry point.

Package root: `com.example.B2BProyect`

## Architecture

**Layered pattern:** Controller → Service → Repository. Each layer lives in its own Maven module.

**DTOs:** Split into `dto/request/` (input) and `dto/response/` (output). Mapping is done via constructors — response DTOs have constructors that accept entities or use JPQL `new` expressions in `@Query` annotations. No MapStruct.

**Auditing:** All auditable entities extend `AuditableEntity` (MappedSuperclass), which adds `createdDate`, `modifiedDate`, `createdBy`, `modifiedBy`, and `version`. The auditor is pulled from the Spring Security context; falls back to `"SYSTEM"` if unauthenticated. JPA auditing is enabled via `@EnableJpaAuditing` on the main class.

**Security:** Stateless JWT via JJWT 0.12.6. The `JwtTokenFilter` extracts the Bearer token and populates `SecurityContext`. Public endpoints: `/api/v1/auth/login`, `/swagger-ui/**`, `/v3/api-docs/**`, `/ws/**`. Method-level security is enabled (`@EnableMethodSecurity(securedEnabled = true)`).

**WebSocket:** STOMP over WebSocket at `/ws` (with SockJS fallback). JWT authentication is validated in `WebSocketAuthChannelInterceptor`. Destinations use `/paymenting` and `/queue` brokers.

**Async logging:** `LogService` uses `@Async("taskLog")` with `REQUIRES_NEW` propagation to persist log entries without blocking the caller. The executor bean is named `"taskLog"` (configured in `InjectConfiguration`).

**External integration:** `SistemaB2B` (in `core/integracion/`) uses Spring's `RestClient` to call the Stereum payment API. API key and base URL are in `application.properties` under `stereum.*`.

**Pagination:** Services accept `Pageable` and return `Page<T>`. Controllers pass `PageRequest.of(page, size, Sort.by(...))` to services.

## Database

PostgreSQL. Schema is managed by Hibernate `ddl-auto=update` — no Flyway/Liquibase. The canonical schema is in `extra/start-scripts/db_schema.sql` and is auto-loaded by Docker Compose on first run.

Local connection: `jdbc:postgresql://localhost:5432/springboot_proyecto`, user `postgres`, password `password`.

Docker DB (via `extra/docker-compose.yaml`): database `b2b_proyecto`, user `owner`, password `123456`.

UUID primary keys are generated by PostgreSQL (`gen_random_uuid()`). Monetary values use `DECIMAL(14,2)`.
